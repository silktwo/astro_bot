# Астро-бот «Astra» — дизайн

**Дата:** 2026-06-17
**Статус:** затверджено до реалізації

## Мета

Окремий Telegram-бот, який щодня шле дівчині користувача персональний
астрологічний прогноз на день, побудований на **двох системах одночасно** —
тропічній і сидеральній. Вона може ставити боту питання в будь-який момент і
кидати йому фото/скріни, які він враховує в контексті.

## Ключові рішення (зафіксовані)

| Рішення | Вибір |
|---|---|
| Платформа | Telegram, **окремий бот** (ID `8660523286`), незалежний від Hermes-асистента |
| Реалізація | Власний легкий Python-сервіс у Docker (`restart: unless-stopped`), деплой через Portainer на Umbrel-сервері (192.168.1.18) |
| Астро-рушій | **Реальні ефемериди** через `pyswisseph` (Swiss Ephemeris). LLM лише інтерпретує готові числа, ніколи не рахує позиції планет |
| Модель — reasoning | `qwen3.7-max` (аналіз: які транзити сьогодні важливі) |
| Модель — текст прогнозу | `claude-sonnet-4.6` (теплий фінальний текст) |
| Модель — vision | `gemini-3.5-flash` (аналіз фото/скрінів) |
| LLM-ендпоінт | opencode-go (`https://opencode.ai/zen/go/v1`, OpenAI-сумісний) — та сама підписка, що в Hermes |
| Мова прогнозу | українська |
| Онбординг | при першому `/start` бот тепло (не анкетою) питає дату/час/місто народження; адмін-команда дозволяє користувачу ввести її карту наперед |

## Чому НЕ всередині Hermes і НЕ n8n

- **Hermes** тримає рівно один `TELEGRAM_BOT_TOKEN` — мультибот не підтримується.
  Окремий бот у тому ж gateway зробити не можна. Другий Hermes-інстанс — надмірно
  важкий заради одного бота.
- **n8n** — для склеювання сервісів тригерами, а не для діалогової логіки з
  реальною математикою (ефемериди), конвеєром двох моделей і станом діалогу.
  Звівся б до обгортки навколо того ж Python, але без контролю.

## Компоненти

Кожен модуль має одну відповідальність, чіткий інтерфейс і тестується окремо.

### 1. `ephemeris.py`
- **Що робить:** рахує позиції планет, аспекти і транзити до натальної карти.
- **Вхід:** натальні дані (дата, час, місто → широта/довгота) + цільова дата.
- **Вихід:** структурований JSON — позиції в **тропічній і сидеральній** системах,
  активні транзити дня, аспекти.
- **Залежності:** `pyswisseph`, дані ефемерид Swiss Ephemeris.
- **Тести:** проти відомих позицій (валідація проти онлайн-ефемерид на кілька дат).
- LLM тут не бере участі.

### 2. `llm.py`
- **Що робить:** тонкий клієнт до opencode-go (OpenAI-сумісний chat completions).
- **Інтерфейс:** `reason(prompt) → qwen3.7-max`, `write(prompt) → claude-sonnet-4.6`,
  `see(image, prompt) → gemini-3.5-flash`.
- **Залежності:** API-ключ opencode-go (env), `httpx`/`openai` SDK.

### 3. `forecast.py`
- **Що робить:** конвеєр щоденного прогнозу.
- **Потік:** натал з `store` → `ephemeris.py` (транзити дня) →
  `llm.reason()` (виділити 2–3 ключові теми дня з обох систем) →
  `llm.write()` (теплий укр. текст з поясненням тропіка vs сидерика) → текст для Telegram.

### 4. `bot.py`
- **Що робить:** хендлери Telegram (python-telegram-bot).
- **Функції:**
  - `/start` → онбординг (питає дату/час/місто або вітає, якщо карта вже введена).
  - Щоденний пуш (через scheduler).
  - Вільні питання → відповідь у контексті її карти + транзитів дня.
  - Прийом фото/скрінів → `llm.see()` → вплести в контекст.
  - Адмін-команда (тільки користувач) — ввести/оновити її натальні дані наперед.

### 5. `store.py`
- **Що робить:** SQLite-персистенція.
- **Дані:** натальна карта, профіль, історія діалогу (контекст відповідей).
- **Том:** монтований Docker-volume, переживає рестарти/ребути.

### 6. Інфраструктура
- `Dockerfile` (python:3.11-slim + pyswisseph + ephemeris-дані).
- `docker-compose.yml` — `restart: unless-stopped`, volume для `natal.db`, env для токена і API-ключа.
- Деплой через Portainer на Umbrel. Автостарт після ребуту — політикою рестарту Docker.

## Потоки даних

**Щоденний прогноз:** scheduler (ранок) → `forecast.py` → Telegram-пуш.

**Питання:** її повідомлення → `bot.py` → контекст (натал+транзити+історія) →
`llm.reason`/`write` → відповідь.

**Фото:** її зображення → `bot.py` → `llm.see` (gemini) → вплести в наступну відповідь.

**Онбординг:** `/start` без карти → діалог-збір даних → `ephemeris` будує натал → `store`.

## Обробка помилок

- **opencode-go недоступний / rate limit:** ретраї з backoff; для щоденного пушу —
  fallback на `qwen3.7-plus`, щоб прогноз усе одно пішов.
- **Ephemeris-помилка:** логувати, не слати «битий» прогноз; повідомити користувача (адміна).
- **Невалідні дані народження при онбордингу:** перепитати конкретне поле.
- **Telegram API збій:** черга/ретрай відправки.

## Тестування

- `ephemeris.py` — юніт-тести проти відомих ефемерид (критично: це джерело правди).
- `llm.py` — мок ендпоінта; перевірка маршрутизації модель↔роль.
- `forecast.py` — інтеграційний тест конвеєра на фіксованій карті/даті (snapshot).
- `bot.py` — тести хендлерів на мок-апдейтах Telegram.

## Зафіксовані рішення (раніше відкриті)

1. **API-ключ opencode-go:** окремий Zen-ключ, виданий саме для цього бота.
   Передається ВИКЛЮЧНО через змінну оточення `OPENCODE_API_KEY` у `.env` на сервері.
   НЕ потрапляє в код, спек чи git. Рекомендовано перевипустити ключ після першого запуску.
2. **Година пушу:** **18:00 за Europe/Warsaw** (scheduler у цьому tz).
3. **Дані ефемерид:** **вбудовані в Docker-образ** — стандартні файли Swiss Ephemeris
   (офіційні, astro.com), завантажуються на етапі білду образу.
